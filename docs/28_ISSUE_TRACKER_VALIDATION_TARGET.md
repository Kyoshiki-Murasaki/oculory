# 28 — Issue-Tracker Validation Target (Phase 5)

## What this is

A **third** validation target for Oculory, after the task server (docs/01–22) and the
sandboxed filesystem server (docs/26). It is a **local, in-memory, deterministic issue
tracker** that simulates a GitHub / Linear-style workflow so we can test the pipeline
against a *richer, stateful* MCP-like server: entity resolution, state transitions,
ambiguous references, invalid transitions, read-only vs. write behaviour, and adversarial
rejection.

### What this is NOT (read this first — no overclaiming)

- It is **not** a real GitHub / Linear / Jira integration. Every "issue" lives in this
  process' memory and is discarded after each recording session.
- It is **not** production MCP-ecosystem validation. There is no network, no external
  server, no MCP SDK wire traffic in the scripted path.
- It is **not** market validation and **not** a security certification.
- Scripted traffic is generated by **deterministic stand-in policies**, not a model. It
  validates the *pipeline mechanics*; it does not prove how a real model behaves. Live
  model runs are a separate, budget-capped, opt-in step (see docs/29).

It reuses the exact same trace → verify → mine → review → approve → suite → replay →
compare pipeline as the other two targets; only the server, scenarios, entities,
verifier, and induced regressions are issue-tracker-specific. Everything additive — no
existing behaviour changed.

## The server (`src/examples/issuetracker/server.ts`)

State model per issue: `id` (ISSUE-N, creation order), `title`, `body`, `status`
(`open` | `closed`), `assignee` (a known user or null), `priority`
(`low` | `normal` | `high`, default `normal`), `labels[]` (sorted), `comments[]`.

- **Known users** (closed allow-list): `alice`, `bob`, `carla`.
- **Allowed labels** (closed allow-list): `bug`, `feature`, `urgent`, `docs`.
- **Statuses**: `open`, `closed`.
- Every scenario runs in a **fresh deterministic fixture** (`fixtures.ts`,
  `issue-tracker-v1`): six seeded issues; a fresh `IssueTrackerServer` is built per
  recording session, so no scenario shares mutable state.

Deterministic search facts the scenarios rely on (title substring, case-insensitive):
`login` → ISSUE-1, ISSUE-2 (ambiguous); `dark` → ISSUE-3; `onboarding` → ISSUE-5.
Seeded closed issues: ISSUE-4, ISSUE-6.

### Tools (9)

| Tool | Args | Behaviour / rejection |
|---|---|---|
| `create_issue` | title, body, priority? | opens a new `open` issue, returns it (with assigned id) |
| `read_issue` | id | returns the issue · `NOT_FOUND` if absent |
| `search_issues` | query | issues whose title contains the query (case-insensitive) |
| `assign_issue` | id, assignee | assigns to a known user · `INVALID_USER` otherwise |
| `label_issue` | id, label | applies an allowed label · `INVALID_LABEL` otherwise |
| `comment_issue` | id, body | appends a comment, preserving existing ones |
| `close_issue` | id | `open`→`closed` · `INVALID_STATE` if already closed |
| `reopen_issue` | id | `closed`→`open` · `INVALID_STATE` if already open |
| `list_issues` | status? | all issues, optionally filtered by status |

Structured error codes: `NOT_FOUND`, `INVALID_USER`, `INVALID_LABEL`, `INVALID_STATE`,
`INVALID_ARGUMENT`. **Policy decision:** closing an already-closed issue is a
**valid rejection** (`INVALID_STATE`), *not* a silent idempotent no-op — this is defined
clearly so the `already_closed_policy_changed` regression has a crisp ground truth.

## Scenario families (`scenarios.ts`, 32 scenarios)

- **Mining** (13): `issue_create` ×2, `issue_read`, `issue_search`, `issue_assign` ×2,
  `issue_label` ×2, `issue_comment`, `issue_close` ×2, `issue_reopen`, `issue_list`.
  Two scenarios for create/assign/label/close give the miner *entity diversity* so it
  binds args to `@entity:*` instead of incidental constants.
- **Holdout** (11): the same families with different ids / users / labels / titles, plus
  two deterministic multi-step cases — `issue_search_read` (search→read) and
  `issue_search_close` (search→close). Holdout is **never mined** (leakage isolation).
- **Adversarial** (6): `issue_missing_id` (NOT_FOUND), `issue_ambiguous_title`
  (two "login" matches → must not mutate), `issue_invalid_user` (INVALID_USER),
  `issue_invalid_label` (INVALID_LABEL), `issue_already_closed` (INVALID_STATE),
  `issue_search_readonly` (search only → no mutation).
- **Smoke** (2): `issue_smoke_list`, `issue_smoke_read`.

## Deterministic verifier (`verifier.ts`)

`verifyIssueOutcome` labels each trace `verified_success` / `valid_rejection` /
`partial_success` / `invalid_acceptance` / `verified_failure` / `unknown`, exactly as the
task and fs verifiers do — "success" means the intended *state / retrieval outcome*
occurred, never merely that a tool returned ok. Condition kinds (`evalIssueCondition`):
`issue_exists`, `issue_absent`, `issue_status`, `issue_assignee`, `issue_priority`,
`label_present`, `comment_present`, `state_unchanged`, `no_new_issue`, `read_consistent`,
`search_consistent`, `list_consistent`, plus the `expected_error` check for adversarial
scenarios. Postconditions cover every requirement: exact tool, args present/equal to
scenario entities, max call count, no unexpected error, expected error for adversarial
cases, state unchanged for read/search-only cases, and the state postconditions (issue
exists / status / assignee / label present / comment present / not-found & invalid handled
safely).

## Candidate mining + risk gating (`mine.ts`, shared `candidate-risk.ts`)

The generic miner already understands `status` / `assignee` / `priority` fields keyed by
an `id` / `title` selector, so scalar issue postconditions fall out for free. `mine.ts`
adds the array-valued postconditions the generic miner can't express (`label_present`,
`comment_present`), emitted only when they hold across every usable trace *and* generalise
to an intent entity. Assertions bind to `@entity:id` / `@entity:assignee` / `@entity:label`
/ `@entity:body` rather than freezing literals. Risk gating (via `annotateCandidates`)
marks as **advisory-only / risky** exactly as the other targets do: adversarial-only
candidates, incidental-constant args, ambiguous cases, and low-support candidates — none
are auto-approved in a live/model run.

## Induced regressions (`mutations.ts`, 11)

10 meaningful defects + 1 benign false-positive probe (`tool_order_changed`), all on the
real server code path:

`close_noop`, `assign_wrong_user`, `label_wrong_issue`, `comment_wrong_issue`,
`search_returns_partial_wrong_match`, `missing_id_succeeds`, `invalid_user_allowed`,
`invalid_label_allowed`, `already_closed_policy_changed`, `readonly_search_mutates_state`,
and `tool_order_changed` (benign).

## Metrics (see docs/29 for the current numbers)

`issue-experiment` reports: traces, outcome counts, families mined, candidate count,
stable-assertion count, unmutated suite pass rate, per-mutation detection (mined suite,
golden checks, schema-smoke proxy), mined precision/recall, and unique detections beyond a
naive schema-smoke baseline — the same pre-registered decision rule as docs/05.

## CLI

```
issue-inspect · issue-scenarios · issue-mutate
issue-smoke · issue-experiment · issue-report
issue-model-smoke · issue-model-experiment · issue-model-replay   (need OPENAI_API_KEY)
```

The server-agnostic `review` / `approve` / `suite` commands accept `--run-dir <dir>` for
issue runs too. Live model runs write only inside `.oculory/runs-live/<run-id>/`
(prefixed `issue-…`), never into the scripted store.
