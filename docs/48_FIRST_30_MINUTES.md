# Your first 30 minutes with Oculory v0.1

This walkthrough uses only the bundled scripted agent, toy MCP server, and disposable Git fixture. It makes no model-provider call, needs no API key, and does not use the statistical miner or an internal scenario ID.

Oculory v0.1 is a release candidate, not an npm publication. Start in a clean source checkout.

> **Platform preflight:** Public v0.1 execution is supported on macOS and Linux only. On Windows, `record`, `replay`, and `demo` fail before Oculory starts any workspace, agent, or MCP child process because descendant cleanup cannot yet be proven. Run this walkthrough in WSL2, Linux, or macOS; native Windows execution requires a Job Object or an equivalent bounded process-tree mechanism.

## Install the local candidate

Run each command from the repository root:

```sh
npm ci
npm run build
npm run typecheck
npm pack
npm install -g ./oculory-0.1.0.tgz
oculory --version
oculory doctor
```

Expected version output is `0.1.0`. `doctor` reports local runtime prerequisites; it does not read provider credential values.
Public `doctor --model` is rejected before any provider environment variable is inspected. The legacy provider-presence diagnostic is available only through the explicitly internal `oculory advanced doctor --model <name>` surface and is not part of this walkthrough.

## See the contradiction

```sh
oculory demo
```

The demo performs the full product story in a disposable directory:

1. record the scripted baseline agent through the real stdio path;
2. draft an editable YAML contract from observed state;
3. replay the clean baseline;
4. replay the deliberately changed scripted agent;
5. render the claim, tool result, and observed-state contradiction;
6. prove process and temporary-directory cleanup.

The changed agent reports success after creating the branch from the wrong base and leaving work staged. Oculory must show the successful claim and tool result without treating either as proof of state.

The final lines are:

```text
Try it on your own task:
  oculory record ./task.yaml
```

If the demo needs a network connection, provider key, account, configuration file, or more than five minutes, the release candidate is blocked.

## Exercise the public commands

The tracked demo task is under `fixtures/demo` and uses provider-free profiles named `baseline` and `changed`. A reference contract sits beside it for fixture and Action validation, but this walkthrough deliberately replays the contract generated by `approve`.

Start the loop with a clean local store:

```sh
oculory record ./fixtures/demo/task.yaml
```

The command prints a friendly run ID and the exact approval command. In a clean store, the first run is `run_0001`:

```sh
oculory approve run_0001 --yes
```

Approval writes a readable YAML file at `oculory.contracts/demo-create-feature-branch.yaml`, a nonignored location intended for Git tracking. It drafts outcome assertions from this one run and does not claim those observations are statistically stable. Review this approved draft before replay.

Replay the baseline through the approved contract and task registration created by the commands above:

```sh
oculory replay --model baseline
```

The baseline must satisfy the configured threshold and exit `0`.

Now replay the changed agent:

```sh
oculory replay --model changed
```

The changed profile must render a behavioral violation and exit `2`. Infrastructure failure uses exit `3`; it is never counted as a behavioral pass.

## Reconstruct the saved diff

The changed replay prints an exact `Diff` command. Run that printed command; it has this shape:

```sh
oculory show <changed-run-id> --diff
```

`show` rebuilds the human view from saved evidence without running the agent again. A missing claim is printed as `claim unavailable`. Ambiguous tool attribution is printed as `no uniquely attributable tool result`.

For automation, pass an explicit report path:

```sh
oculory replay --model baseline --report ./oculory-report.json
```

The JSON report is a complete artifact, not terminal text wrapped in JSON.

## Try your own task

Copy [examples/v0.1/task.yaml](../examples/v0.1/task.yaml), then replace only:

- the prompt;
- the agent profile argv and environment allowlist;
- the MCP server executable and argv;
- the workspace strategy;
- target adapter configuration and watch scope.

Task and contract files accept literal argv arrays only. Do not put a shell command string in either file. Keep credentials in explicitly named environment variables; never place values in YAML.

The normal loop remains:

```sh
oculory record ./task.yaml
oculory approve <run-id>
oculory replay --model <profile>
```

Read [the contract reference](49_V0_1_CONTRACT_REFERENCE.md) before editing assertions and [the adapter reference](50_V0_1_ADAPTER_REFERENCE.md) before widening a watch scope.

## What this walkthrough proves

It proves only that the bundled provider-free fixture can exercise the local v0.1 record, approve, replay, renderer, package, and cleanup path on the tested environment. It does not establish real-model reliability, production Postgres or GitHub behavior, production readiness, security certification, adoption, usability, retention, demand, or willingness to pay.

Native Windows is intentionally outside this execution claim. The platform preflight runs before disposable workspace setup and before any agent or MCP process is spawned; schema inspection and saved-run rendering do not broaden the execution claim.
