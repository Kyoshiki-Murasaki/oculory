# 01 — Product definition

**One sentence.** Oculory converts recorded MCP agent-tool sessions into stable, provenance-linked, deterministic-first regression tests that detect meaningful behaviour changes after modifications to a server, tool schema, prompt, model, or environment.

**Problem.** MCP servers sit between non-deterministic agents and real side effects. Teams change schemas, descriptions, and backends without any test that asks "can agents still accomplish the tasks users actually perform?" Existing checks are schema-level (does the tool answer?) rather than outcome-level (did the task actually happen in the environment?).

**Intended user.** A developer who builds or maintains an MCP server and wants a CI gate against agent-visible regressions.

**Core workflow.** `record → verify → normalize → mine → review → approve → suite → run → compare → report`, all local, all via one CLI (docs/09).

**Jobs to be done.** (1) Know when a server change breaks agent workflows before users do. (2) Turn "it worked yesterday" into an executable, reviewable artifact. (3) Catch lying servers — success responses without the corresponding state change.

**Inputs.** Agent-tool session traffic against an MCP server (currently generated by scripted policies against the bundled demo server; model providers and external trace import are extension points). **Outputs.** JSONL traces, outcome records, candidate tests (JSON review table), a versioned approved suite, run results, comparison reports (JSON + Markdown).

## Lifecycles
- **Trace:** recorded (raw JSONL, validated) → outcome-verified → normalised (entities + redaction) → immutable input to mining or holdout evaluation.
- **Candidate test:** mined per scenario family from verified traces only → status `candidate` → human review → `approved` / `rejected` / `advisory`.
- **Approval:** individual (`oculory approve <id> --reason`), or bulk for stable deterministic assertions (`--all-stable`); every decision is recorded with reason and timestamp inside the artifact.
- **Replay:** fixture reset → scenario re-executed per agent → mined assertions + golden outcome check evaluated deterministically per trial.
- **Regression report:** two runs diffed; a finding is a `regression` only if the test passed at baseline and fails now; exit code 2 for CI.
- **CI:** `oculory run` in a workflow; non-zero exit gates the merge (docs/09 §CI flow).

## Glossary (the distinctions that matter)
| Term | Meaning |
|---|---|
| Raw trace | Verbatim record of one session: tools, args, result digests, per-step state changes, env snapshots before/after |
| Normalised trace | Raw trace + deterministic outcome record + extracted intent entities + redaction applied |
| Outcome label | `verified_success`, `verified_failure`, `partial_success`, `valid_rejection`, `invalid_acceptance`, `unknown` — judged against environment state, never tool status |
| Scenario family | Group of intents that mean the same task in different words; the mining unit and the leakage boundary |
| Candidate test | Miner output: assertions + confidence + provenance, awaiting review |
| Approved test | Candidate a human accepted; only its `stable` assertions compile into the suite |
| Advisory test | Runs and reports but can never fail a build |
| Gate-eligible test | May fail a build once promoted; deterministic assertion types only |
| Blocking test | Gate-eligible test explicitly promoted to block merges |
| Regression | Baseline-pass → current-fail on the same test with identical suite hash |
| Unstable result | Same agent disagreeing with itself across trials; quarantined, never a regression |

## Principles
Local-first · deterministic-first (no LLM judging anywhere in the loop today) · observed behaviour is never automatically ground truth · every assertion carries provenance · benign changes must not fail builds · a suite that is red on an unchanged server is a bug in Oculory, not in the server.

## Success / failure criteria
Success: the pre-registered experiment decision rule (docs/05) is met, unmutated pass rate is 100%, and a stranger can run the quick start unaided. Failure: false positives on benign mutations, unstable assertions reaching gates, or mined suites that only re-detect what schema diffing already catches.

## Non-goals (MVP)
Web frontend, dashboards, accounts/teams/billing, enterprise auth, cloud/multi-tenant anything, leaderboards, certification/registry scoring, cross-client browser automation, destructive production-tool execution, generic observability, arbitrary protocols, plugin marketplace, visual branding.
